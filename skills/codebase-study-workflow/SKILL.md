# Codebase Study Workflow

Teach an existing repository from the whole system down to functional modules, real runtime flows, learning steps, and individual symbols. Default to read-only analysis. The objective is a durable mental model, not a file-by-file narration or a full rewrite of a large codebase.

Read `references/artifact-templates.md` when creating or updating study artifacts.

## Scope Boundary

Use this skill when the user primarily wants to understand code that already exists. The code may be legacy, generated by AI, inherited, partially written, or currently being debugged.

Use `$feature-study-workflow` when the primary goal is to turn requirements into a new feature. If a feature task is blocked by unfamiliar code, study only the affected module here, then return to the feature workflow with a concise handoff.

## Non-Negotiable Distinctions

Keep these three layers visibly separate:

1. **Verified code facts**: supported by source, configuration, tests, language navigation, or runtime evidence.
2. **Inferred teaching order**: an AI-designed sequence for learning or reconstructing the final code. It is not the repository's actual Git history or the author's original implementation order.
3. **Optional practice implementation**: a small analogous lab or an approved micro-change. It is not evidence about how the original system was built.

Label uncertainty as `verified`, `inferred`, or `unknown`. Never turn a plausible import, name, or directory layout into a claimed runtime path without corroboration.

## Core Contract

- Start with a shallow system map, then go deep into one selected functional module or golden flow.
- Decompose as `system -> functional module -> runtime flow -> learning step -> symbols/concepts`.
- Prefer user-visible capabilities and domain responsibilities over technical-layer tours such as “all controllers” or “all models.”
- Explain actual execution in order: entry -> dispatch/calls -> data and state changes -> side effects/failures -> result.
- For each important function or variable, show its definition, responsibility, readers/callers, writes/callees, lifecycle, and change impact.
- Prefer a runnable learning step, but do not fake independence when the code cannot run in isolation. Every step must end as one of `runnable`, `testable`, `verifiable`, or `inspectable` with a concrete check.
- For huge repositories, map broadly and study selectively. Never propose reproducing tens of thousands of lines just to understand them.
- Persist the mental model in structured Chinese notes unless the user requests another language.
- Stop at module boundaries so the user can recall, choose depth, and change learning track.

## Default Permissions

Analysis is read-only by default. Reading files, searching references, inspecting history, running safe existing checks, and starting an existing program for observation are allowed when relevant.

Before creating a learning lab, editing production code, changing tests, or adding repository documents outside an already accepted study directory, explain the intended files and ask for explicit confirmation. A request to “explain” or “analyze” does not authorize source changes.

## Repository Evidence Method

Use repository-native evidence before teaching:

- inspect manifests, build/run commands, entry points, routes, jobs, event handlers, dependency injection, persistence, and external boundaries;
- search definitions and references with repository search and language-aware navigation when available;
- read focused tests as executable behavior evidence;
- run the smallest safe existing command that resolves an important uncertainty;
- use Git history only when the user asks about actual evolution or when history materially verifies intent;
- cite file paths and symbols in notes so claims can be revisited.

Do not infer runtime order from directory order. Do not assume every reference is executed. Distinguish static possibility from an observed or test-supported path.

## Study Artifacts And File Structure

Use the project's documentation convention when one exists. Otherwise keep all notes under:

```text
docs/codebase-study/<study-slug>/
  index.md
  progress.md
  system-map.md
  pedagogical-roadmap.md
  modules/
    <module-slug>.md
  symbol-index.md
  concept-notes.md
  learning-log.md
```

Create module files only when that module is actually studied. Do not scatter explanations beside arbitrary source files.

- `index.md`: scope, navigation, and how the documents relate.
- `progress.md`: current module, flow, learning track, evidence gaps, mastery, and next action.
- `system-map.md`: boundaries, entry points, dependencies, state/data, and module inventory.
- `pedagogical-roadmap.md`: the inferred reconstruction order with outcome classes and checks.
- `modules/<module>.md`: one module's responsibilities, flows, code mapping, and recall checks.
- `symbol-index.md`: durable lookup for important functions, types, variables, and where they are used.
- `concept-notes.md`: syntax, language, framework, and architectural concepts linked back to real code.
- `learning-log.md`: decisions, user explanations, corrections, and spaced recall prompts; never a chat transcript.

Before any lab, inspect existing `examples`, `samples`, `playground`, or tutorial conventions. If none exists, propose `learning-labs/<study-slug>/` as an isolated location and ask for confirmation. Never mix simplified lab code into production modules.

## Progress Queries

When asked for status:

1. find the most recently modified active `docs/codebase-study/*/progress.md`;
2. report current scope, module, flow, track, verified versus inferred state, mastered items, evidence gaps, waiting party, and next action;
3. do not continue analysis or implementation unless requested.

If multiple studies are active, list them. If none exists, offer to initialize one.

## Phase 1: Establish Scope And Learner Context

Determine what the user needs to understand: the entire system at map depth, a subsystem, a runtime behavior, a change-impact question, or specific unfamiliar code. Ask only questions that change the study route, such as language familiarity, desired depth, and whether the repository can be run safely.

Do not demand a complete inventory from the user. Inspect the repository first and record scope, exclusions, and unanswered questions.

## Phase 2: Build A Shallow System Map

Inspect enough of the repository to explain:

- what the system does and who invokes it;
- how it starts, builds, tests, and is configured;
- top-level directories and their actual responsibilities;
- major functional modules and dependency direction;
- key entry points, state stores, data models, and external systems;
- existing file-placement, test, docs, and example conventions;
- important unknowns and evidence needed to resolve them.

For each claim, mark `verified`, `inferred`, or `unknown` and link evidence. On a large repository, use breadth-first sampling: manifests, entry points, module roots, representative tests, and one golden path. Do not deeply read every file.

Create or update `system-map.md`, then present the module map and stop for the user's selection.

## Phase 3: Decompose Functional Modules And Golden Flows

For each relevant functional module, record:

- user/system responsibility and boundaries;
- entry points and observable results;
- upstream/downstream modules;
- important state and external effects;
- representative runtime flows;
- confidence and remaining evidence gaps;
- recommended study priority.

Choose flows that connect several responsibilities and reveal the module's design. Avoid a flat list of files. For very large modules, split by independent capability or golden flow.

Stop after the map. The user chooses which module or flow to study first.

## Phase 4: Derive A Pedagogical Reconstruction Roadmap

For the selected module, derive an AI-authored teaching order from the final code. State prominently:

> This is an inferred pedagogical reconstruction order, not verified implementation history.

Each roadmap step includes:

- observable or conceptual outcome;
- code responsibilities and relevant symbols;
- prerequisites;
- outcome class: `runnable`, `testable`, `verifiable`, or `inspectable`;
- exact validation or inspection method;
- syntax, framework, or design concepts introduced;
- what later step depends on it.

Use the strongest practical outcome class:

1. `runnable`: a user-visible or system-visible slice can execute.
2. `testable`: focused existing tests or a safe harness demonstrate behavior.
3. `verifiable`: static checks, type checks, reference tracing, or deterministic assertions confirm the step.
4. `inspectable`: a coherent responsibility or contract can only be examined in context; give precise evidence and dependency explanation.

Never split merely to make many steps. Some behavior becomes meaningful only after several functions are present; group those functions into one coherent step and explain why it cannot be smaller.

## Phase 5: Walk Through One Module In Runtime Order

Default to a read-only walkthrough. Follow one real flow from entry to outcome:

1. show the entry trigger and inputs;
2. trace calls and dispatch with evidence;
3. trace data transformations, state reads/writes, and ownership;
4. show external effects, failures, cleanup, and retries;
5. connect the final result to tests or observable behavior;
6. summarize why the module is shaped this way and what would break if it changed.

Teach only relevant detail. When unfamiliar syntax or APIs appear, create a concept note that includes the real code location, plain-language meaning, why it is used here, a minimal example when useful, and where it appears again.

For an important symbol, record a symbol card:

- definition and responsibility;
- inputs, outputs, and mutable state;
- callers/readers and callees/writers;
- creation and lifetime;
- where it appears in the current runtime flow;
- likely change impact;
- confidence and evidence.

Do not explain isolated symbols without reconnecting them to the module and runtime flow.

At natural checkpoints, ask the user for a short recall: predict the next call, explain why a responsibility exists, or identify the impact of a small change. Correct one misconception at a time and update notes.

## Phase 6: Choose The Next Learning Track

After one module or golden flow, summarize verified behavior, inferred teaching order, concepts learned, remaining unknowns, and recall status. Then stop and offer three explicit tracks:

### A. Continue The Read-Only Walkthrough

Study the next flow or module using the same evidence method. This is the default for very large repositories.

### B. Build A Simplified Analogous Lab

Use when implementation practice would materially improve understanding.

- Recreate only the essential mechanism, not the original repository.
- Keep it independent and small enough to finish.
- Create a correspondence table: original responsibility/symbol -> lab counterpart -> intentionally omitted complexity.
- State that the lab is an analogy, not production code or historical reconstruction.
- Explain proposed file placement and ask before creating it.

### C. Make A Safe Micro-Change In The Original Repository

Use only when the user wants transfer practice and the change is small, reversible, and testable.

- Ask the user to predict affected flow, symbols, and checks.
- Inspect ownership and place changes according to existing conventions.
- Describe the exact change and validation, then obtain explicit approval.
- Implement only the approved scope and compare predicted versus actual impact.

Do not turn either implementation track into a full codebase rewrite.

## Phase 7: Consolidate And Resume

At a module boundary:

- update the system map only with newly verified facts;
- update the pedagogical roadmap when evidence changes the teaching order;
- add important symbols and concepts to their indexes;
- record the user's explanation, corrections, and spaced-recall prompts;
- mark mastery per flow, not per file count;
- identify the smallest next module or flow that extends the mental model.

When resuming after a long gap, begin with `progress.md`, the current module note, and a short recall prompt. Do not restart the repository tour unless the map is stale or the user asks.

## Completion Criteria

A study scope is complete when:

- the requested system/module map is supported by evidence;
- selected runtime flows can be explained from entry to result;
- important symbols are connected to their usage and lifecycle;
- the pedagogical roadmap is clearly labeled and has checkable steps;
- the user can explain the main responsibility and one change impact;
- remaining unknowns and out-of-scope modules are explicit.

Completion never requires reimplementing the entire repository.
